iOS Design Pattern

  • Design Pattern
  • Objective-C
  • iOS

写过那么多的代码，也在很多场景使用不同的设计模式，MVC、MVVM、MVP等。其本质就是让代码变得更容易修改，编写，测试（你真的会写代码来测试自己的代码么？）。

这片文章并不会从理论上说各个模式，而是从实践中，体会各个设计模式的优缺点。

MVC

MVP

对于这个模式，用的最多的就是在视频聊天、直播场景中使用。但是我觉得依旧有很多不足的地方。

  • 分出的presenter如何和主控制器进行交互？
    • 弱引用
      • 优点
        • 直接和主控制器进行互动
      • 缺点
        • 直接访问主控制器意味着可以间接访问其他presenter，导致代码的混乱，一个业务代码可能分散在各个presenter中。
        • 各个presenter的界限明显降低。
    • Delegate
      • 优点
        • 将主控制器和presenter分开，限制了presenter随意调用主控制器相关内容。
      • 缺点
        • 各个presenter交互成了问题。大量的delegate带来的是主控制器依旧很重。

Plugins Design Pattern

对于开发者来说，理想的开发方式就是插件化开发，开发一个功能，添加该插件，就可以使用，不用的时候去掉插件。多么理想的开发，插件可以脱离主体进行测试。

Summery

对于设计模式，整个项目不可能采用同一个设计模式进行开发。不同的场景使用不同的设计模式也会使项目便于维护。

相比于设计模式，再好的设计模式也不能阻止开发者随意的代码编写，每个设计模式都有优点和缺点，不能指望一个完美的设计模式来解决所有的问题。

代码越来越混乱的原因就是设计模式已经更不上业务逻辑的改变（你愿意为你的业务逻辑重构代码么？）。当别人接手你的代码的时候发现很难改，但是又需要完成产品提出的任务，终于好不容易改好了，Bug测试完了，根本不愿意再重构代码，因为你怕又出现Bug，继续陷入无止境的加班中。

想要代码变得整洁，并不是是要一下子写出完美的代码，而是在感觉到危机的时候能够及时的阻止代码结构的进一步恶化（你能做到么？）。
